Looking at the code carefully, **the bug is still present**. Let me trace through exactly why.

**The intended flow:**
1. User inactive 15+ min → token expires → popup → redirect to `/` → session cleared

**What actually happens:**

In `client.ts`, when a 401 hits:
```js
if (errorCode === "UNAUTHORIZED") {
  const refreshed = await attemptRefresh()
  if (refreshed) {
    response = await makeRequest() // silently continues — no popup
  } else {
    await expireSession("inactivity") // only fires if refresh FAILS
  }
}
```

And `attemptRefresh()` calls `/auth/refresh` which in `auth_service.py` accepts expired tokens via `decode_token(allow_expired=True)` and issues a fresh one **as long as the session is under 8 hours**:

```python
if datetime.now(timezone.utc) - created > timedelta(hours=8):
    return {"ok": False, "error": "SESSION_EXPIRED"}
# otherwise → issues new token
```

So after 15 min inactivity, the token expires → next API call gets a 401 → refresh silently succeeds → user continues with no popup. **The popup only triggers after the 8-hour absolute max**, not after 15 minutes.

**There's also a second issue** — a duplicate modal between `App.tsx` and `routes.tsx`:

```jsx
// App.tsx — OUTSIDE RouterProvider
export default function App() {
  return (
    <AuthProvider>
      <RouterProvider router={router} />
      <SessionExpiredOverlay /> {/* useNavigate() will fail here — no router context */}
    </AuthProvider>
  )
}

// routes.tsx — correctly inside router
function RootLayout() {
  return (
    <>
      <Outlet />
      {sessionExpiredReason && <SessionExpiredModal ... />} {/* works */}
    </>
  )
}
```

`SessionExpiredModal` calls `useNavigate()`, which will throw when rendered from `App.tsx` since it's outside the router context.

**To actually fix the 15-min inactivity behavior**, the refresh endpoint should refuse to refresh an already-expired token (rather than accepting it), or the client should not attempt a refresh on token expiry and instead show the popup immediately. The `attemptRefresh()` call in the `UNAUTHORIZED` branch is what prevents the intended behavior.
